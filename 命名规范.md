# 命名规范

集中管理 AI-qskills 生态下所有命名规则。

---

## 一、通用规则

适用于所有命名场景（Skill、博客、脚本、目录等）：

| # | 规则 | 说明 | 示例 |
|---|------|------|------|
| 1 | **以域开头** | 先确定所属域，以域前缀开头，保证同类聚集 | `opencode-`、`record-`、`AI-`、`script-` |
| 2 | **英文优先** | 用英文，不用拼音（专有名词如 Feishu 除外） | `record-router` ✅ `jilu-luyouqi` ❌ |
| 3 | **见名知意 + 简洁** | 看到名字就知道是什么，不冗余 | `record-to-skill` ✅ `skill-ji-lu-dao-skill` ❌ |
| 4 | **连接符按场景遵循惯例** | 不同场景使用不同连接符，详见各章节 | |

命名时如需连接多个单词，用中划线 `-` 还是下划线 `_`，取决于该场景的业界惯例，不强行统一。具体约定如下：

| 中文名 | 连接符 | 适用场景 | 示例 |
|--------|--------|----------|------|
| 中划线 | `-` | 目录名 | Skill 目录名、Repo 名、博客分类目录 |
| 中划线 | `-` | 文件名 | 博客文件名 |
| 下划线 | `_` | 文件名 | Shell 脚本文件名、Python 文件名 |

## 二、Skill 命名规则

### 结构

```
<域>-<对象>-<行为>
```

各部分均为小写英文 kebab-case。

> **域前缀的含义**：每个域前缀限定了该技能的**适用范围**。例如 `opencode-` 表示「专为 opencode 设计，不适合其他场景复用」；`record-` 表示「属于归档体系的一部分」。命名前先确认域的适用范围，不要给通用工具套上特定域的前缀。

### 示例

| 名字 | 域 | 对象 | 行为 |
|------|----|------|------|
| `opencode-sessions-manager` | opencode | sessions | manager |
| `record-to-skill` | record | (to) skill | (路由到) |
| `script-specification` | script | specification | (创建) |

### 命名案例分析

为什么这样设计？以下用真实案例说明命名背后的原因。

#### 案例一：`opencode-sessions-manager`

```
opencode-sessions-manager
~~~~~~~~ ~~~~~~~~ ~~~~~~~
域        对象      行为
（专供   管理     管理器
opencode 会话
使用）
```

**为什么用 `opencode-` 域？**

因为这个工具**只管理 opencode 的会话**（自动记录会话 ID、交互式恢复），离开 opencode 毫无用处。`opencode-` 前缀准确标明了它的适用范围。

如果它是一个通用的 session 管理器（比如管理 tmux、screen、终端标签页），就不能用 `opencode-`，而是应该用 `session-` 或其他通用前缀。

> **教训**：先问"这个工具给谁用、用在哪儿？"确定域；再问"它操作什么？"确定对象；最后问"它做了什么？"确定行为。

#### 案例二：`record-to-skill` — "动词"域的命名

```
record-to-skill
~~~~~~~ ~~~~~~~~
域      行为+对象
```

**为什么 `record-` 是一个域？**

因为这个技能的定位是"记录"体系中的一个环节，与 `record-router`、`record-to-hexo-blog`、`record-to-script-repo` 组成一个完整的归档工作流：

```
record-router             ← 路由分发
  ├── record-to-skill     ← 存档为 Skill
  ├── record-to-hexo-blog ← 存档为博客
  └── record-to-script-repo ← 存档为脚本
```

所有 `record-*` 技能天然按字母聚集，一眼看清归档体系。如果用 `skill-recorder`、`blog-recorder`、`script-recorder`，它们就会散落在目录的不同位置，看不出关联。

#### 案例三：`script-specification` — 域+对象即可

```
script-specification
~~~~~~ ~~~~~~~~~~~~~
域      对象
```

如果这个技能只是"创建符合规范的脚本"，不涉及更复杂的行为（发布、测试等），那么 `域+对象` 就够了，不需要再加行为词。简洁本身就是设计目标。

命名时反问自己：
1. **域是什么？** → opencode/record/script/life...
2. **对象是什么？** → sessions/skill/blog/script/reply...
3. **还需要行为词吗？** → manager/creator/router...
4. **去掉行为词够清晰吗？** → 够就不加

---

### 如何判断新技能应该用哪个域前缀？

答：看它的**定位和用途**，而不是随便找一个像的：

```
开 opencode 相关工具          → opencode-
归档/记录/路由类              → record-
脚本创建/发布/测试            → script-
生活/娱乐/回复                → life-
内容整理/代码组织              → organize-
UI/UX 设计                    → ui-ux-
业务方案（营商环境等）        → business-
```

如果不确定，参考已有同类技能的前缀，或用**通用规则**中的"以域开头"反问：如果所有技能按字母排序，我希望这个技能跟谁排在一起？

## 三、博客文章命名规则

### 文件路径

```
分类目录/文章名.md
```

### 文章名格式

```
<域前缀>-<序号><文章名>
```

- **域前缀**：与 Skill 通用规则一致的域，如 `AI-`、`AI Tool-`
- **序号**：同一级下按已有文件递增，用 `①、②、③...` 或 `1、2、3...`
- **文章名**：中文，简洁说明文章内容

示例：
- `AI/AI-③opencode会话管理.md`
- `AI/AI Tool-②cc-connect.md`

### hexo frontmatter

```yaml
---
title: 文章标题
date: 2026-05-18 16:59:48
categories: 
- AI            # 一级分类
tags:
- AI           # 标签
---

[toc]          # 如需要目录
```

### 图片资源

根据 `post_asset_folder: true` 配置，每篇文章的图片放在同名的**文件夹**中：

```
AI/AI-③opencode会话管理.md
AI/AI-③opencode会话管理/截图1.png
```

## 四、博客目录分类规则

### `总目录.md` 条目格式

```markdown
  - [第N节：文章标题](分类/文章名)
```

示例：
```markdown
  - [第3节：opencode会话管理](AI/AI-③opencode会话管理)
```

### 新增条目

1. 确定所属分类（参考已有分类体系）
2. 按同级序号递增
3. 在对应分类列表末尾新增一行

## 五、脚本库目录规则

### 文件名

Shell 和 Python 脚本统一用 `_`（snake_case）：

```
get_system_env.sh    ✅
upload_to_all_and_log.sh  ✅
sign_apk_tool.py     ✅
env_var_1get_by_manual.sh ✅

get-system-env.sh    ❌（除非已有大量此类文件）
```

### 目录名

遵循仓库既有惯例，不强行统一：
- `script-qbase` / `script-branch-json-file` 内以 `_` 为主（`env_variables/`、`path_util/`）
- `AI-qskills` 内以 `-` 为主（`record-to-skill/`）

### Repo 名

统一用 `-`（kebab-case）：

```
script-branch-json-file ✅
script-qbase           ✅
```
