# 博客文章评分

对 Hexo 博客文章进行多维度评分，用于首页排序、内容质量评估。

## 触发条件

- "给文章打分" — 对某分类下的文章进行评分
- "建立评分系统" — 初始化评分标准
- "评分" — 对某个路径下的文章进行评估
- "评估文章质量" — 查看或更新文章评分

## 交互流程

### Step 1: 检查当前目录

检查当前工作目录，确定目标路径。

### Step 2: 询问用户

根据当前目录，询问：

```
当前目录: /Users/lichaoqian/Project/CQBook/dvlproadHexo/source/_posts/xxx
是否对该目录下的文章进行评分？
├── yes         → 使用当前目录
├── quit / q    → 退出
└── <其他路径>  → 视为用户输入的路径，进入 Step 3
```

### Step 3: 确认路径

用户输入其他路径时，显示路径并要求确认：

```
确认对以下路径下的所有文章进行评分？
路径: /Users/xxx/xxx
[yes / no]
```

### Step 4: 执行评分

1. 递归查找该目录下所有 `.md` 文件
2. 逐篇阅读文件内容
3. 按评分标准对每篇文章打分
4. 输出评分结果文件

### Step 5: 输出评分结果

评分结果保存为 **`rating_目录名.json`**，放在 `总目录/data/` 下。

```json
[
  {
    "path": "Architecture架构/框架相关/网络相关/Client Cache Best Practice.md",
    "title": "Client Cache Best Practice",
    "total": 86.8,
    "scores": {
      "depth": 85,
      "practical": 88,
      "completeness": 90,
      "originality": 75,
      "timeliness": 93,
      "code_quality": 90
    }
  }
]
```

## 评分标准

### 总分公式

```
最终评分 = 内容深度×30% + 实用价值×20% + 完整性×30% + 原创性×10% + 时效性×5% + 代码质量×5%
```

每个维度 0~100 分，最终评分 0~100 分。

### ① 内容深度（权重 30%）

| 分数 | 标准 |
|------|------|
| 90~100 | 源码级分析，讲清原理/设计模式/架构思想，有深度对比 |
| 70~89 | 讲清实现原理，有核心源码/流程图，不仅停留在 API 使用 |
| 50~69 | 有基本原理说明，但偏重使用步骤 |
| 30~49 | 主要罗列步骤/配置，缺少原理分析 |
| 0~29 | 纯资料搬运或简单操作记录 |

**注意：** 方案方向对比类文章不应要求源码级分析。不同技术路线的设计思想对比本身就有深度价值。

### ② 实用价值（权重 20%）

| 分数 | 标准 |
|------|------|
| 90~100 | 日常高频问题，方案可直接复用，省大量排查时间 |
| 70~89 | 常见场景的完整解决方案，稍加调整即可用 |
| 50~69 | 有用但场景较窄，或方案偏基础 |
| 30~49 | 偶尔能用，或已有更好替代方案 |
| 0~29 | 实际使用价值低 |

**注意：** 列出历史方案作为对比参考是正常的，不应因某个备选方案过时扣除整篇的实用分。

### ③ 完整性（权重 30%）

| 分数 | 标准 |
|------|------|
| 90~100 | 背景→原理→实践→踩坑→总结，结构完整，逻辑闭环 |
| 70~89 | 有原理有实践有总结，少数环节略简 |
| 50~69 | 有基本结构但缺少总结或实践验证 |
| 30~49 | 内容零散，结构不完整 |
| 0~29 | 只是片段笔记 |

**注意：** AI 看不到图片内容，评估含图片的文章时应意识到此局限，适当调分。

### ④ 原创性（权重 10%）

| 分数 | 标准 |
|------|------|
| 90~100 | 独立思考/踩坑经验/自创方法论，非网上常见内容 |
| 70~89 | 在官方文档/他人基础上做了整合提炼，有自己的理解 |
| 50~69 | 编译整理为主，少量个人补充 |
| 30~49 | 大段引用或翻译，个人贡献较少 |
| 0~29 | 明显搬运 |

**注意：** 原创性应看核心内容是否原创，而非外部引用数量。引用多本身是严谨的表现。

### ⑤ 时效性（权重 5%）

| 分数 | 标准 |
|------|------|
| 90~100 | 经典/常青内容，方案至今仍为主流或最佳实践 |
| 70~89 | 仍可参考，核心思想/方案不过时 |
| 50~69 | 部分方案已过时，但思路仍有借鉴意义 |
| 30~49 | 技术已明显迭代，仅作为历史参考 |
| 0~29 | 已废弃/被完全替代，不推荐参考 |

**辅助分类：内容保鲜期速查表**

| 内容类别 | 基准时效 | 说明 | 典型文章 |
|---------|:-------:|------|---------|
| **基础原理类** 🔥 | 90~95 | 基本理论不变，几乎不朽 | HTTP、设计模式、数据结构、算法 |
| **方法论/流程类** ✅ | 85~90 | 方法逻辑长久有效 | 技术选型流程、CodeReview规范、分支规范 |
| **架构思想类** 📐 | 75~85 | 核心思想持久，具体实现会迭代 | 组件化、MVC/MVVM、面向协议编程 |
| **框架/工具类** ⚡ | 30~60 | 具体框架容易被替代 | CTMediator、SDWebImage、GetX |
| **项目记录类** 📝 | 20~40 | 时效性最短，只对当时有用 | BUGLY崩溃汇总、特定版本踩坑 |

**使用方式：** 先看文章属于哪类 → 取基准分 → 根据内容质量和原创程度上下调整 ±5~10 分。

例如：组件化文章 → 架构思想类（基准80）→ 内容原创且实践充分 → **80** ✓

### ⑥ 代码质量（权重 5%）

| 分数 | 标准 |
|------|------|
| 90~100 | 代码完整可运行，有命名规范、健壮性处理 |
| 70~89 | 代码正确，结构清晰，少量细节可以优化 |
| 50~69 | 代码有示意性，但缺少完整性或规范 |
| 30~49 | 代码片段不完整，缺少上下文 |
| 0~29 | 无代码或代码有明显错误 |

**注意：** 伪代码只要能清晰表达核心逻辑即可，重点在于传达思想而非通过编译。

### ⑦ 篇幅调整（微调，不占权重）

篇幅不独立评分，作为**同等质量下的微调**。当在其他维度上犹豫时，可以往高给。

| 行数 | 判定 | 微调幅度 |
|:----:|------|:--------:|
| 400~800 | 中等篇幅 | +1 |
| 800~1200 | 长文 | +2 |
| 1200+ | 超长篇 | +3 |

注意：这是**置信度加分**（篇幅长 ≈ 投入精力多），不是自动加分。

### ⑧ 评分示例

**场景**：一篇 iOS 源码分析文章

| 维度 | 得分 | 权重 | 加权得分 |
|------|:---:|:----:|:--------:|
| 内容深度 | 85 | 30% | 25.50 |
| 实用价值 | 75 | 20% | 15.00 |
| 完整性 | 90 | 30% | 27.00 |
| 原创性 | 70 | 10% | 7.00 |
| 时效性 | 80 | 5% | 4.00 |
| 代码质量 | 85 | 5% | 4.25 |
| **总分** | | | **82.75** |

## 评分案例参考

### 案例：框架设计模式-⑦组件化

| 维度 | 初分 | 终分 | 关键结论 |
|------|:---:|:---:|---------|
| 内容深度 | 75→**85** | 方案方向对比不需要源码级分析，设计思想对比本身有深度价值 |
| 实用价值 | 75→**80** | 列出历史方案作对比是正常的，不应因备选过时扣分 |
| 完整性 | 75→**80** | AI 看不到图片内容，需预留余量 |
| 原创性 | 75→**85** | 核心代码是原创，外部引用仅为次要补充 |
| 代码质量 | 70→**80** | 伪代码表达清楚核心逻辑即可 |

**结论启示：**
- 内容深度评分要区分"方案方向对比"和"单个方案源码分析"，前者不应要求源码级分析
- 原创性评估应看核心内容是否原创，而不是外部引用数量
- 代码质量的本质是"能否表达清楚"，而非"能否编译运行"
- AI 看不到图片内容，评估含图片的文章时应意识到此局限
