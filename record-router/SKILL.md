# Record Router — 内容分发路由

判断待记录的内容类型，路由到对应的 Skill 去处理细节。本身不做具体写入操作。

## 触发条件

- "记录这个"
- "保存这个"  
- "归档这个"
- "把这个记下来"
- "把这个写到博客" / "记录到博客"
- "创建skill" / "完善我的skill"
- "把这个脚本入库" / "放到脚本库"
- 完成工具/脚本/方案后，AI 主动询问用户是否要记录归档

## 设计说明：为什么要绕 router 一圈

所有 `record-to-*` 类 Skill 的入口统一走 router，即使"创建skill"这种明确命令也不例外：

```
用户说"创建skill"
        │
        ▼
  record-router（判断内容是否有"多重属性"）
        │
        ├─ 纯工具/脚本 → 只路由到 record-to-skill
        └─ 工具+知识   → 询问是否同时写博客 + 建 Skill
```

**不直接走 record-to-skill 的原因**：
- 一个工具（如 `opencode-sessions-manager`）同时包含**可执行部分**（脚本、安装步骤）和**知识部分**（技术原理、DB 结构、设计决策）
- record-to-skill 只关注"怎么做"，看不到"这部分也应写博客"
- router 站在全局视角，能发现内容的多重属性，避免做完 Skill 才后知后觉

> 口诀：**明确命令绕一圈，router 帮你看看有没有别的方向。**

## 知识库 — 你的归档目的地

| 类型 | 目标 | 对应 Skill |
|------|------|-----------|
| 知识/经验/踩坑记录 | hexo 博客 `_posts/*` | record-to-hexo-blog |
| 可复用的工具/脚本/工作流 | AI-qskills 下的 Skill | record-to-skill |
| 可被 `qbase` 或 `qtool` 调用的脚本 | script-qbase / script-branch-json-file | record-to-script-repo |

## 执行流程

### 1. 分析内容类型

首先判断用户想记录的内容属于哪一类：

- **知识/经验类**：踩坑记录、实现思路、技术决策、使用技巧 → 走博客
- **工具/脚本/工作流类**：可复用的自动化方案、CLI 包装、配置流程 → 走 Skill
- **脚本类**：可被 `qbase` 或 `qtool` 调用的独立脚本 → 走脚本库
- **综合类**（既是知识又是工具）→ 询问用户优先级，或按顺序全做

> **何时建议创建 Skill**：在完成以下事项时，即使没要求记录，也应主动询问是否要创建对应 Skill：
> - **工具/脚本类**：完成了一个可复用的 Shell 脚本、工具函数或工作流（如 `source_opencode.sh`）
> - **博客文章类**：写完一篇记录某个工具/工作流用法的博文
> - **重复性工作类**：用户多次执行类似的繁琐操作，可以自动化
>
> 提示语："这篇内容/这个工具是否要创建为 Skill？以后新机器加载 Skill 就能自动配置，不用翻文档。"

### 2. 询问用户确认

```
AI 分析出内容类型后：
"这个内容看起来属于 [类型]，我建议：
  1. [处理方式 A]
  2. [处理方式 B]
  您想怎么处理？"
```

### 3. 路由到对应 Skill

当目标 Skill 涉及分类判断、命名约定或流程对比时，优先参考以下规范：

- 分类归属规则：[《分类规范.md》](../分类规范.md)
- 命名约定：[《命名规范.md》](../命名规范.md)
- 流程对比：[《流程对比规范.md》](../流程对比规范.md)
- **路径引用规则**：提及非本项目中的文件（Skill、规范文档等）时使用 HTTP 链接，禁止用本地绝对路径。
  - 同一仓库下的其他目录可使用相对路径
  - 其他仓库统一用 `https://github.com/<owner>/<repo>/blob/main/<path>` 格式

根据用户选择，执行对应 Skill 的处理流程：

```
record-router
  ├── 写博客     → 引用 record-to-hexo-blog 的规范
  ├── 创建 Skill → 引用 record-to-skill 的创建流程
  └── 入库脚本   → 引用 record-to-script-repo 的规范
```

### 4. 完成后询问

每个环节完成后，询问用户是否还需要其他处理。
