# Record to Hexo Blog — 博客写入规范

将知识/经验/踩坑记录写入 hexo 博客，按规范匹配合适的分类和文件名。

## 触发条件

- **被 record-router 路由"写博客"任务时** → 走写作流程
- **用户输入"整理博客" / "博客归类" / "移动博客文章" 等明确含"博客"关键词的指令时** → 走归类整理流程

## 博客结构参考

### 根目录

```
~/Project/dvlproadHexo/source/_posts/
```

每次写文章前，先按分类规则确定归属，再在 `总目录.md` 中找到对应位置。主要分类：

| 顶级分类 | 示例子分类 |
|---------|-----------|
| Architecture架构 | 架构相关、框架相关、基础规范、监控相关 |
| 实用工具 | Terminal、Hexo、GitBook、Jenkins、Nginx |
| Script | Shell、Ruby、JavaScript、Python |
| AI | ChatGPT、AIGC、AI Agent、AI Tool |
| iOS部分 | 开发规范、Swift、性能相关、第三方库 |
| Flutter部分 | 入门、集成、交互、详解、进阶 |

## 写作流程（A）

→ A1: 分类判断 → A2: 新建文件 → A3: 写入 front matter → A4: 添加 more 截断 → A5: 新增总目录 → A6: 目录索引 README → A7: 独立 HTML 文件（如有）→ A8: 用户确认

## 归类整理流程（B）

→ B1: 分类判断 → B2: 移动/重命名 → B3: 更新 front matter → B4: 添加 more 截断 → B5: 修改总目录 → B6: 目录索引 README → B7: 独立 HTML 文件（如有）→ B8: 用户确认

## 流程细则

[A1/B1: 分类判断] → 读素材/源文件，对照分类规范确定归属，不确定时询问用户
[A2: 新建文件] → 按分类下已有文件序号递增，如 AI Agent 下已有 ① ② ③ 则下一篇为 AI-④xxx
[B2: 移动/重命名] → 按新路径移动文件，涉及删除源文件时先向用户确认
[A3: 写入 front matter] → 写入 title、date、categories，与文件路径一致
[B3: 更新 front matter] → 修改 categories 匹配新路径，文件名改变时同步修改 title
[A4/B4: 添加 more 截断] → 在文章第一个章节结束后插入 `<!-- more -->`，让首页只显示摘要。已有则跳过
[A5: 新增总目录条目] → 在对应分类列表末尾按格式新增
[B5: 修改总目录条目] → 更新对应分类下的条目路径
[A6/B6: 目录索引 README] → 目标目录已有 README 则更新内容列表；无 README 但 ≥2 篇时询问用户是否创建
  - 文件名: `目录名_README.md`
  - 内容: 内容链接列表 + 待补充提醒
  - 总目录: 新增或更新 `* [前言](目录/目录名_README)` 条目
[A7/B7: 独立 HTML 文件] → 文章可能需要附带独立 HTML 页面（如 demo、项目列表），需要确保它被 Hexo 正确输出。按 normalize-blog-style 的《Step 6: 独立 HTML 文件处理》执行
[A8/B8: 用户确认] → 展示结果让用户确认，确认后再结束

## 规范细则

### 撰写风格
- 简洁，附代码/命令
- 记录核心知识和关键要点
- 提及 Skill 路径时使用 HTTP 链接，禁止用本地绝对路径。
  - 如引用 AI-qskills 中的 Skill，格式：[AI-qskills 中的 xxx](https://github.com/dvlproad/AI-qskills/blob/main/xxx/SKILL.md)
  - 如引用其他仓库中的 Skill，使用该仓库对应的 HTTP 链接
- Skill 引用放在读者理解「方案是什么」之后、文章收尾之前。读者先了解问题与方案，再得知去哪里获取，自然引导行动。不要放在开头（读者还没理解背景）或末尾收尾后才出现
- 不写无意义的废话

### Front matter 规则
- `date` 从 Git 首次提交获取
- `updated` 省略（首次新建时 date = updated）
- `categories` 与文件路径一致
- 图片资源使用 hexo `post_asset_folder: true` 机制，图片放在同名文件夹下

### 首页摘要控制

`<!-- more -->` 是 Hexo 支持的截断标记，写在文章正文中。首页只显示标记之前的内容作为摘要，全文才展开。

- 位置：第一个正文章节（通常是问题背景/简介）结束后，下一个标题之前
- 整理旧文章时：检查内容中是否有 `<!-- more -->`，没有则按上述规则补加
- 少数文章可能需要手动调整，用户确认时负责检查

### 总目录条目格式
```markdown
  - [第N节：文章标题](分类/文章名)
```
